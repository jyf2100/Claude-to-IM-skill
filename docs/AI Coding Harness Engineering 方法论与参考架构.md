# AI Coding Harness Engineering 方法论与参考架构

## 定义

在 AI Coding 领域，`Harness engineering` 指的是：

> 构建一套让 coding agent 能稳定输出标准化、可验证、可交付工程结果的执行与验收系统。

这里的重点不是单次生成能力，而是**稳定交付能力**。它关注的不是“模型能不能写出一段代码”，而是“agent 能不能像工程团队里的合格成员一样，持续输出可审核、可测试、可接手的标准交付物”。

## 核心目标

一个成熟的 AI coding harness，应该解决以下问题：

- 将自然语言需求转化为可执行任务合同
- 约束 agent 的执行流程、权限边界和工作范围
- 强制产出符合团队规范的标准交付物
- 自动验证改动是否满足验收标准
- 对同类任务持续回归，衡量交付稳定性

换句话说，harness 解决的是从：

`Prompt -> 看起来能用的代码`

升级到：

`任务输入 -> 规范执行 -> 自动验证 -> 标准交付`

## 与普通 Eval 的区别

普通评测更关注：

- 输出答案是否正确
- benchmark 分数是否提高
- 单次任务表现是否足够好

AI coding harness 更关注：

- 任务理解是否准确
- 改动范围是否受控
- 是否遵守仓库规则和本地约定
- 是否完成必要验证
- 是否生成标准化交付说明
- 是否能在多次任务中保持稳定

因此，AI coding 语境下的 harness engineering 更接近：

- Delivery engineering
- Agent reliability engineering
- Eval infrastructure engineering
- 工程化执行系统设计

## 标准交付物是什么

在 AI coding 场景里，标准交付物通常不只是代码本身，还可能包括：

- 代码变更
- 测试补充或修复
- 文档更新
- 配置改动说明
- 验证结果
- 风险与限制说明
- 最终交付摘要

一个高质量 harness 的目标，是让 agent 输出一个**可审查、可验证、可接手的交付包**，而不是一段“看起来完成了”的回答。

## 方法论

### 1. Task Contract：任务合同化

将模糊需求转换为可执行合同。至少应包括：

- 任务目标
- 范围边界
- 非目标事项
- 验收标准
- 输出格式
- 约束条件

示例：

```yaml
task:
  goal: 修复登录页面按钮点击无响应问题
  scope:
    - src/login.tsx
    - src/hooks/useAuth.ts
  non_goals:
    - 不重构认证流程
    - 不修改 UI 视觉样式
  acceptance:
    - 按钮点击后可以触发表单提交
    - 相关测试通过
    - 不影响已有登录错误提示
  deliverables:
    - 最小代码改动
    - 验证结果摘要
    - 风险说明
```

### 2. Workflow：流程固定化

要求 agent 按固定阶段执行，而不是自由发挥：

1. 理解任务
2. 读取规则
3. 定位相关代码
4. 制定计划
5. 实施修改
6. 运行验证
7. 生成交付说明

流程固定化的收益：

- 便于观察 agent 是否漏步骤
- 便于在失败时定位是哪一阶段出了问题
- 便于替换模型而不替换流程

### 3. Constraints：环境与权限约束

对 agent 的工作环境进行硬约束，包括：

- 文件系统可读写范围
- 网络权限
- 命令白名单或黑名单
- 是否允许安装依赖
- 是否允许改动无关文件
- 是否允许执行危险操作

这部分的目标是把 agent 从“强能力体”变成“受控工程工位”。

### 4. Validation：自动验收化

把“人类 reviewer 会检查的事情”尽可能前移到机器验收：

- 单元测试
- 集成测试
- 类型检查
- lint
- build
- 输出格式校验
- 文件改动边界检查
- 文档遗漏检查

验收系统越完备，agent 输出就越接近真实团队可接收的交付标准。

### 5. Delivery Format：输出标准化

要求 agent 的最终交付遵循统一结构，例如：

- 改了什么
- 为什么这样改
- 修改了哪些文件
- 跑了哪些验证
- 结果是否通过
- 还存在哪些风险
- 建议下一步做什么

这会显著提升结果的可读性、可审查性和系统可集成性。

### 6. Regression：持续回归化

对历史任务集持续重复执行，跟踪：

- 任务完成率
- 一次验收通过率
- 返工率
- 非目标改动率
- 最终交付合规率

这一步决定 harness 是否真的把 agent 变成了稳定交付系统，而不仅是一次性 demo。

## 五个关键对象

### 1. Task Contract

描述任务到底要做什么，以及什么算完成。

### 2. Execution Trace

记录 agent 的执行步骤、命令、文件读写和关键决策，用于调试、审计和复现。

### 3. Artifact Bundle

最终交付包，通常包含代码改动、测试结果、说明文档和风险提示。

### 4. Validation Report

对验收流程的结构化记录，明确哪些检查通过、哪些失败、失败原因是什么。

### 5. Failure Taxonomy

失败分类体系，用于分析问题根因，例如：

- 需求理解错误
- 修改范围越界
- 忽略仓库规则
- 没有完成验证
- 输出格式不合规
- 工具调用失败后恢复不当

## 典型流水线

一个实际可运行的 AI coding harness 可以按如下流程工作：

1. 接收用户需求或工单
2. 生成 `Task Contract`
3. 读取仓库规则和局部上下文
4. 约束可修改范围和可执行动作
5. 生成计划
6. 执行改动
7. 运行最小必要验证
8. 检查是否存在越界改动或遗漏项
9. 生成标准化交付说明
10. 记录评测数据并进入回归数据集

## 核心指标

在 AI coding harness 中，比模型分数更重要的是交付指标：

- **任务完成率**：任务是否真正完成
- **一次通过率**：是否无需返工即可验收
- **验证通过率**：测试、lint、build 等检查通过比例
- **交付合规率**：最终输出是否符合模板和要求
- **非目标改动率**：是否产生与任务无关的修改
- **返工轮次**：完成任务需要多少次修正
- **平均交付时长**：从接收任务到交付完成耗时
- **Reviewer 接受率**：人工审核者对交付结果的接受度

## 常见失败模式

### 输入层失败

- 需求含糊，未被转成明确任务合同
- 验收标准缺失，导致 agent 自行假设

### 执行层失败

- 未先读取仓库规范
- 直接修改大量无关文件
- 命令执行策略过宽，产生副作用

### 验收层失败

- 代码改完但没跑测试
- 测试通过，但遗漏了类型检查或 build
- 只验证 happy path，未覆盖关键边界

### 输出层失败

- 没有交代修改内容
- 没有说明验证结果
- 没有披露已知风险

### 运营层失败

- 没有失败归因体系
- 没有历史回归集
- 只看平均结果，不看关键任务是否失守

## 一个最小可行 Harness（MVP）

如果从零开始搭建，建议先实现以下最小版本：

### 输入

- 一个统一的任务输入格式
- 明确的 `Task Contract` 模板

### 执行

- 一个 runner，能驱动 agent 读文件、改文件、跑命令
- 一个基本 plan 机制
- 基础权限约束

### 验收

- 支持运行最小测试集
- 支持检查最终输出模板
- 支持检查是否发生越界修改

### 输出

- 固定最终报告模板
- 记录改动文件和验证结果

### 运营

- 保存任务输入、输出和执行轨迹
- 支持复跑历史任务

## 实用公式

可以用一个简化公式来记住 AI coding harness 的本质：

`Harness = Contract + Workflow + Constraints + Validation + Delivery Format`

少任何一项，系统都容易退化成“偶尔写得不错，但不稳定交付”。

## 结论

在 AI Coding 领域，Harness engineering 的本质不是提升模型的“表达能力”，而是保障 agent 的“交付下限”。

它真正创造的价值是：

- 让任务从模糊需求变成可执行合同
- 让执行从自由生成变成受控流程
- 让结果从文本回答变成标准交付物
- 让质量从主观判断变成自动验收
- 让系统从单次 demo 变成可持续运营的工程能力

因此，一个成熟的 AI coding harness，本质上是在为 agent 建造一个可控、可审查、可回归、可度量的工程工位。
